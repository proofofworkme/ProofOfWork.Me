# Repository Hygiene

Every approved ProofOfWork.Me update ends with a repository-hygiene pass, even
when no commit is created. Every commit is a durable handoff to the next human
or agent.

This workflow keeps current documentation, operating memory, generated output,
and local build state aligned without treating age alone as proof that a file is
stale.

## Required Handoff

Before final staging or committing:

1. Re-read the staged diff and identify every behavior, protocol, product,
   infrastructure, deployment, and operating-memory consequence.
2. Reconcile `SOUL.md` and every affected canonical document:
   `README.md`, `PROOFOFWORK_IDS.md`, `MARKETPLACE.md`,
   `OP_RETURN_INFRASTRUCTURE.md`, and `MAIL_ORGANIZATION.md`.
3. Review the classified note inventory in `repository-hygiene.json`. Correct
   contradictory current-state claims, broken local links, abandoned drafts,
   superseded screenshots or snapshots, stale generated output, and orphaned
   files when the evidence is conclusive.
4. Run `npm run hygiene:fix` to remove only allowlisted rebuildable local
   state.
5. Run the tests appropriate to the update, then run
   `npm run hygiene:check`.
6. Inspect `git status --short`, `git status --short --ignored`, and the final
   staged diff. Nothing unrelated, secret, local, or rebuildable belongs in the
   commit.
7. Commit with the required review trailers:

   ```text
   Documentation-Impact: updated
   Repository-Hygiene: reviewed
   ```

   Use `Documentation-Impact: reviewed-no-change` only after the semantic
   review finds every relevant document already accurate. When a tracked file
   is intentionally deleted, add one exact trailer per deleted path:

   ```text
   Repository-Hygiene-Removed: path/to/removed-file
   ```

   A protected ledger, refund snapshot, public/release/deploy asset, or
   intentionally tracked output also requires prior explicit user approval and
   a second exact attestation:

   ```text
   Repository-Hygiene-Protected-Removal: path/to/protected-file
   ```

The tracked `pre-commit`, `prepare-commit-msg`, and `commit-msg` hooks enforce
the mechanical part of this contract, including amended and merge commits. CI
validates the complete repository state and trailers of every commit in a push
or pull request. Hooks do not replace semantic judgment.

## `SOUL.md`

`SOUL.md` is operating memory, not an append-only diary and not the protocol
specification.

- Update it when a change creates durable product memory, agent posture,
  product voice, or a lasting operating invariant.
- Reconcile or remove obsolete undated current-state claims.
- Preserve dated launch/archive history unless it is factually wrong; later
  rules should explicitly supersede historical rules rather than silently
  rewriting the record.
- Do not manufacture a no-op edit merely to touch the file. A reviewed,
  already-accurate `SOUL.md` is a valid result.
- Protocol documentation, source code, confirmed chain truth, and explicit
  user instructions remain authoritative when they conflict with `SOUL.md`.

## Note Classes

`repository-hygiene.json` classifies every tracked Markdown, MDC, and Cursor
rules file. A new note must be classified in the same commit. The classes
distinguish:

- canonical current product/protocol/operating documentation;
- agent bootstrap instructions;
- protected ledgers and audit evidence;
- staged or future protocol research;
- presentation/generated notes; and
- supporting documentation.

Classification does not make a note correct forever. It makes its purpose
explicit so the next update knows whether to reconcile, preserve, regenerate,
or retire it.

Declared deterministic generators are executed in a temporary directory during
the check. The tracked general deck copies and ProofOfWork Computer model
outputs must byte-match their current sources; removing a generator from the
manifest is itself rejected as a policy downgrade. CI repeats this check on
the supported Node.js LTS majors so generated bytes cannot drift with
runtime-specific floating-point formatting.

## Automatic Cleanup Boundary

`npm run hygiene:fix` may remove only the exact allowlist in
`repository-hygiene.json`, currently:

- `dist/`;
- root `.vite/` and `.pow-api-cache/`;
- Vite caches inside `node_modules/`, but never `node_modules/` itself;
- coverage and browser-test report directories; and
- root npm/debug development logs matching the configured patterns.

The cleaner verifies that a target is inside the repository, ignored,
untracked, not a symlink, and does not cross a filesystem or mounted-volume
boundary before deleting it. It rechecks those boundaries during removal and
never invokes `git clean`. The executable contains the same exact cleanup
ceiling as the configuration, so a staged policy edit can neither expand nor
disable cleanup.

## Protected And Judgment-Required Material

Old is not stale. Never delete, rewrite, or regenerate these merely because
newer behavior exists:

- confirmed protocol and chain history;
- compatibility, migration, replay, recovery, and rollback notes;
- txids and block-bound evidence;
- refund, treasury, and bug-bounty ledgers;
- canonical audit fixtures and refund snapshots;
- intentionally tracked `output/`, `public/`, or deployment assets;
- local `.env` files, wallet material, keys, or private fundraising files; or
- branches and worktrees that may contain unmerged commits.

Tracked deletion outside the already approved update remains a scope expansion.
Stop and ask before doing it. The standing hygiene approval covers only the
allowlisted rebuildable local state and semantic reconciliation within the
approved update.

## Hook Installation

The current checkout uses the tracked `.githooks/` directory:

```bash
npm run hooks:install
```

`npm install` installs it automatically only when the checkout has no custom
`core.hooksPath`. An existing custom hook path is never overwritten silently.
The CI check remains the clone-independent backstop.
